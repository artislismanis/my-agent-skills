# Implementation Plan: Excalidraw Diagram Plugin

**Branch**: `001-excalidraw-diagrams` | **Date**: 2026-03-21 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-excalidraw-diagrams/spec.md`

## Summary

Create the first marketplace plugin — an Excalidraw diagramming skill that
enables Claude to generate professional, consistently styled diagrams. The
plugin bundles markdown reference material (format spec, styling config, diagram
templates), a `SKILL.md` that instructs Claude how to generate valid Excalidraw
JSON, and a self-contained Node render script for PNG validation. Delivered as a
standard plugin package under `plugins/diagramming/excalidraw-diagrams/`.

## Technical Context

**Language/Version**: Markdown (reference material) + Node.js 22+ (render script only)
**Primary Dependencies**: `excalidraw-to-svg` (JSON→SVG), `@resvg/resvg-js` (SVG→PNG)
**Storage**: N/A — file-based output (JSON written to disk, PNG rendered on demand)
**Testing**: Manual validation via render script (generate JSON → render PNG → visual check); JSON schema validation for format correctness
**Target Platform**: Claude Code plugin (cross-platform, any OS with Node.js)
**Project Type**: Claude plugin (skill package)
**Performance Goals**: PNG render < 30 seconds for typical diagrams (5–30 elements)
**Constraints**: Self-contained script with deps declared in local package.json, no global installs, no browser automation, no MCP server
**Scale/Scope**: Single-user local execution; diagrams typically 5–50 elements

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

### I. Spec-First Development ✅

Specification (`spec.md`) was written and validated via quality checklist before
this plan. Clarification phase found no critical ambiguities. Plan follows spec.

### II. Test-First Implementation ✅ (with interpretation)

The plugin is primarily markdown reference material — not executable code in the
traditional sense. The render script is the only executable component. For the
render script: tests will be written before implementation (input JSON → expected
PNG output). For skill quality: acceptance scenarios from the spec define
testable criteria (valid JSON structure, correct styling, proper bindings).
Constitution exempts standalone scripts from observability requirements (v1.0.1).

### III. User Story Independence ✅

Each story delivers standalone value:

- P1 (generate diagrams): Works independently with just reference + styling docs
- P2 (PNG render): Works with any valid Excalidraw JSON, independent of templates
- P2 (templates): Enhances generation quality, independent of render script
- P3 (iteration): Builds on P1 reference material, no new components needed

### IV. Incremental Delivery (MVP-First) ✅

P1 (core generation with styling) is fully functional before P2 work begins.
P2 stories (render script, templates) are independent of each other and can be
delivered in either order. P3 (iteration) requires no additional implementation
beyond what P1 provides — it's validated by the reference material's completeness.

### V. Simplicity (YAGNI) ✅

- One colour palette (not configurable)
- Five diagram templates (the requested set, no extras)
- Single render script (not an MCP server or service)
- Markdown references (not a structured database or API)
- No abstraction layers — direct, flat file structure

**Structure Decision**: Single skill within a plugin package. All diagram
templates live under `assets/templates/` as markdown files containing layout
conventions and example JSON patterns. The SKILL.md loads references on demand
and selects the appropriate template based on the diagram type requested.
This follows the agentskills.io folder-based skill format defined in CLAUDE.md.
